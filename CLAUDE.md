# BizDynamite Visual Editor — Complete Technical Documentation

## What Is This Project?

BizDynamite is a platform where users describe a business idea and Claude's API generates a **working MVP web app as a single HTML file** — all HTML, CSS, and JavaScript in one file. The problem: users want to customize the look of the generated app (colors, fonts, layout, text) without touching code. This project is a **visual drag-and-drop editor** that lets them do that using GrapesJS, without breaking the JavaScript functionality.

This `editor.html` file is a standalone proof-of-concept. It will eventually be integrated into a Laravel backend.

---

## File Structure

```
nova3/
  index.html    — The Claude-generated MVP app (the SOURCE file being edited)
  editor.html   — The GrapesJS visual editor (loads and edits index.html)
  CLAUDE.md     — This documentation file
```

### What Is `index.html`?

A 2105-line e-commerce app called "NovaGadget" generated by Claude. It is a fully self-contained single-file app:

| Section | Lines | What's in it |
|---------|-------|-------------|
| `<style>` block | 7–395 (~390 lines) | All CSS: CSS custom properties (variables), layout, responsive breakpoints, animations, modal styles, cart sidebar, hover effects, form styles |
| `<body>` HTML | 396–1149 (~750 lines) | Header with nav, hero section, product grid, product detail modal, auth/login modal, cart sidebar overlay, search overlay, checkout form, footer |
| `<script>` block | 1150–2104 (~950 lines) | Product data array, cart state management, page navigation (`showPage()`), product filtering, modal open/close, cart add/remove, checkout form validation, toast notifications |

**Critical characteristics that make editing hard:**
- **74 inline event handlers** — buttons and links use `onclick="showPage('home')"`, `onclick="addToCart(1)"`, `onclick="closeModal()"`, etc. directly in the HTML
- **87 DOM references in JS** — the JavaScript uses `getElementById()` and `querySelector()` to find and manipulate elements. If those elements are deleted or their IDs are changed, the app breaks
- **CSS variables everywhere** — colors and spacing are defined as `--primary`, `--accent`, etc. in `:root` and referenced throughout

## The Core Problem We Had to Solve

### Why You Can't Just Load HTML into GrapesJS

GrapesJS is designed for building web pages from scratch. When you load existing HTML into it, it does two destructive things:

**1. It strips all inline event handlers.** Every `onclick`, `onsubmit`, `onchange`, `oninput` attribute is removed. GrapesJS treats these as non-visual and discards them during its internal HTML parsing. This means: after editing, every button that said `onclick="addToCart(1)"` now does nothing. Navigation breaks. Modals won't open or close. The cart doesn't work. The app is dead.

**2. `getCss()` only returns CSS that GrapesJS itself created.** If you load 390 lines of CSS into GrapesJS and the user changes one element's color, `getCss()` returns only that one color change — not the other 389 lines. If you use `getCss()` as your final CSS, you lose all original styling: responsive breakpoints, animations, modal positioning, everything.

These two issues meant that a naive "load HTML into GrapesJS, let user edit, export HTML from GrapesJS" approach produces a **completely broken app** — no JavaScript functionality and most CSS missing.

## The Architecture: How We Solved It

### Strategy: Parse → Edit → Reassemble (with the original file as source of truth)

The key insight: **don't let GrapesJS be the source of truth for the final output.** Instead, keep the original `index.html` intact and only use GrapesJS for visual CSS overrides. The original file already has working HTML, CSS, JS, and inline handlers. GrapesJS contributes only the visual changes the user makes.

### The Three Phases in Detail

#### Phase 1: Parse (`boot()` → `parseSourceMeta()` → `renderViaIframe()`)

When the editor loads, the `boot()` function runs. Here's what happens step by step:

**Step 1a: Fetch the source file**
```js
fetch('index.html').then(r => r.text())
```
The raw HTML string is saved into `savedSourceHTML`. This variable is the single source of truth — it is the complete, unmodified original file. It is used verbatim in preview/export later.

**Step 1b: Parse metadata with `parseSourceMeta()`**

This function uses `DOMParser` to parse the HTML string into a DOM tree, then extracts:

- **Scripts** → `savedScripts[]` — Every `<script>` tag is extracted (both `src` external scripts and inline script content). They are stored and then removed from the DOM so GrapesJS never sees them. GrapesJS does not handle JavaScript and would either discard it or cause conflicts.

- **CSS** → `savedOriginalCSS` — Every `<style>` tag's text content is extracted and concatenated. This is the full 390-line CSS block. It gets injected into the GrapesJS canvas later so elements look correct while editing.

- **Head meta** → `savedHeadMeta` — The contents of `<head>` (title, meta tags, etc.) are saved for reassembly.

- **Protected IDs** → `protectedIds[]` — The function joins all script content into one string, then uses regex to find every ID referenced by JavaScript:
  - `/getElementById\s*\(\s*['"]([^'"]+)['"]\s*\)/g` — finds `getElementById('cartSidebar')` → protects `#cartSidebar`
  - `/querySelector(?:All)?\s*\(\s*['"]#([a-zA-Z][\w-]*)['"]\s*\)/g` — finds `querySelector('#productModal')` → protects `#productModal`

- **Protected classes** → `protectedClasses[]` — Same regex approach:
  - `/querySelector(?:All)?\s*\(\s*['"]\.([a-zA-Z][\w-]*)['"]\s*\)/g` — finds `querySelector('.product-card')` → protects elements with class `product-card`

**Step 1c: Render in a hidden iframe with `renderViaIframe()`**

This is the most complex part of parsing. The function creates a hidden iframe, writes the full source HTML into it, and lets the browser execute the JavaScript. Why?

Because the source HTML has JavaScript that **dynamically generates content**. For example, the product grid is populated by JS — the HTML just has an empty `<div id="productGrid">`, and the script fills it with product cards on page load. If we loaded only the static HTML into GrapesJS, the user would see an empty product grid.

The iframe approach lets the JS run, populate the DOM, and then we extract the **fully rendered** HTML (with product cards, populated modals, etc.) to show in GrapesJS.

Here's what happens inside `renderViaIframe()`:

1. Creates an invisible iframe (positioned off-screen at `left: -9999px`)
2. Writes the complete source HTML into it with `iDoc.write(sourceHTML)` — this triggers the browser to parse HTML, apply CSS, and execute JavaScript
3. Waits 600ms (to let `DOMContentLoaded` fire and JS populate the DOM)
4. Removes `<script>` and `<style>` tags from the rendered DOM (GrapesJS doesn't need them)
5. **Maps inline event handlers to `data-biz-id` attributes:**
   - Walks every element in the rendered DOM
   - If an element has any `on*` attribute (onclick, onsubmit, etc.), it:
     - Generates a unique ID: `data-biz-id="biz-0"`, `data-biz-1`, `data-biz-2`, etc.
     - Stores the handler values in `inlineHandlersMap`: `{ "biz-0": { "onclick": "showPage('home')" }, "biz-1": { "onclick": "addToCart(1)" }, ... }`
     - Copies handlers to `data-on*` attributes: `data-onclick="showPage('home')"` (GrapesJS preserves `data-*` attributes)
     - Removes the original `on*` attributes (so GrapesJS doesn't strip them — they're already safely stored)
6. Extracts `iDoc.body.innerHTML` — this is the fully rendered, handler-tagged HTML
7. Removes the iframe from the DOM

**Step 1d: Initialize GrapesJS**

The rendered body HTML is placed into the `#gjs` container via `innerHTML`, and `grapesjs.init()` is called with `fromElement: true` so it reads the content already in the div.

On the `load` event:
- The original CSS (`savedOriginalCSS`) is injected into the GrapesJS canvas iframe as a `<style>` tag, so elements appear with correct styling while editing
- `protectElements()` runs to lock JS-dependent elements (details below)

#### Phase 2: Edit (GrapesJS canvas — what the user interacts with)

The user sees a visual canvas showing the rendered app. They can:
- Click elements to select them
- Change styles via the right panel (colors, fonts, sizes, spacing, borders, backgrounds, effects)
- Drag blocks from the left panel (text, images, columns, forms)
- Switch between Desktop/Tablet/Mobile views
- Navigate between pages (Home, Shop, About, Contact, Checkout) via the top tab bar
- Show/hide overlay elements (Product Modal, Auth Modal, Cart Sidebar, Search Overlay) via the dropdown

Protected elements (ones that JS depends on) can be styled but not deleted or dragged.

#### Phase 3: Reassemble (`reassemble()` — for preview and export)

This is where the "original file as source of truth" strategy pays off. The `reassemble()` function is very simple:

```js
function reassemble() {
    var gjsCSS = gjsEditor.getCss();

    // If no visual edits were made, return the original file unchanged
    if (!gjsCSS || gjsCSS.trim() === '' || gjsCSS.trim() === '* { box-sizing: border-box; } body {margin: 0;}') {
        return savedSourceHTML;
    }

    // Inject GrapesJS CSS overrides into the <head> of the original file
    var overrideStyle = '\n<style>\n/* === Visual editor overrides === */\n' + gjsCSS + '\n</style>';
    return savedSourceHTML.replace('</head>', overrideStyle + '\n</head>');
}
```

That's it. The function:
1. Gets the CSS that GrapesJS created (only the user's visual edits)
2. If there are no edits, returns the **original file unchanged** — all handlers, scripts, CSS intact
3. If there are edits, injects them as a new `<style>` block before `</head>` in the original file

The original file already has all 74 inline handlers, all 390 lines of CSS, and all 950 lines of JavaScript. We don't need to reconstruct anything — we just append visual overrides.

**Why this approach is better than the previous one (which tried to reconstruct the file):**

Earlier, `reassemble()` tried to:
1. Get HTML from `gjsEditor.getHtml()` (which has mangled HTML — stripped handlers, missing SVGs, altered button structure)
2. Get CSS from `gjsEditor.getCss()` (incomplete — missing 95% of original CSS)
3. Re-inject handlers from `inlineHandlersMap` using `data-biz-id` lookups
4. Concatenate original CSS + override CSS
5. Rebuild the `<script>` tags

This was fragile and produced broken output. GrapesJS mangles HTML in ways that are hard to fix:
- SVGs inside buttons get altered or removed
- Button content gets restructured
- Some elements get wrapper divs added
- Elements that JS populates via `innerHTML` (like `#cartItems`, `#cartCount`) lose their dynamic content

The current approach avoids all of these problems by never using GrapesJS HTML output. The original file IS the output, with CSS overrides layered on top.

## Element Protection System

### Why It Exists

The generated app has ~87 places where JavaScript references DOM elements by ID or class. If a user deletes `#cartSidebar` or `#productModal` in the visual editor, the JavaScript will throw errors and the app breaks. The protection system prevents this.

### How It Works

**Step 1: Detection** (happens in `parseSourceMeta()`)

All JavaScript content from the source file is joined into one string. Three regex patterns scan for DOM references:

```
getElementById('cartSidebar')    → adds 'cartSidebar' to protectedIds
querySelector('#productModal')   → adds 'productModal' to protectedIds
querySelector('.product-card')   → adds 'product-card' to protectedClasses
```

Additionally, any element tagged with `data-biz-id` (meaning it has inline event handlers) is also protected.

**Step 2: Locking** (happens in `protectElements()`)

After GrapesJS loads, the function walks every component in the component tree. For each component, it checks:
- Does its `id` appear in `protectedIds`?
- Does any of its classes appear in `protectedClasses`?
- Does it have a `data-biz-id` attribute?

If yes, it sets:
```js
comp.set({
    removable: false,    // can't delete
    draggable: false,    // can't drag/reorder
    copyable: false,     // can't duplicate
    stylable: true,      // CAN still change visual styles
});
comp.set('custom-name', 'LOCK #elementId');  // shows in layers panel
```

**Step 3: Enforcement** (happens on `component:remove` event)

If a protected element is somehow removed (GrapesJS can still allow it in some edge cases), the editor catches the `component:remove` event, checks if `removable === false`, and if so: immediately undoes the action and shows a warning toast.

## Preview and Export

### Preview

The `togglePreview()` function:
1. Calls `reassemble()` to get the complete HTML
2. Creates a `Blob` with type `text/html`
3. Generates a `URL.createObjectURL()` for the blob
4. Opens it in a new browser tab with `window.open(url, '_blank')`

The user sees the fully working app — all JavaScript runs, all handlers work, all CSS is present (original + overrides).

### Export

The `handleExport()` function:
1. Calls `reassemble()` to get the complete HTML
2. Creates a Blob and object URL
3. Creates a hidden `<a>` element with `download="index-edited.html"`
4. Triggers a click to download the file
5. Cleans up the object URL and element

### Save

The `handleSave()` function saves to `localStorage`:
- Key `bizdynamite-project`: `{ html: gjsEditor.getHtml(), css: gjsEditor.getCss(), timestamp }`
- Key `bizdynamite-handlers`: the `inlineHandlersMap` object

On next load, if localStorage data exists, the editor restores the user's previous editing state instead of re-parsing the source file.

## Bugs We Hit and How We Fixed Them

### Bug 1: `SyntaxError: Unexpected end of input`

**What happened:** The editor page was completely blank. Browser console showed `SyntaxError: Unexpected end of input`.

**Why:** The editor's JavaScript contained template literals with HTML strings that included `</script>`. For example:
```js
var html = `<script>alert('hi')</script>`;  // BROKEN
```
The browser's HTML parser doesn't understand JavaScript. When it sees `</script>` anywhere inside a `<script>` block — even inside a string — it thinks the script tag is ending. So the browser splits the JavaScript mid-string, producing a syntax error.

**Fix:** Every occurrence of `</script>`, `</style>`, and similar HTML closing tags inside JavaScript strings is built via string concatenation:
```js
var html = '<' + '/script>';  // WORKS — browser parser doesn't see </script>
```
This pattern is used throughout `editor.html` in the `reassemble()` function and error messages. Look at lines 368-369 and 707-709 to see it in action.

### Bug 2: GrapesJS Shows Default Empty Project Instead of Our HTML

**What happened:** GrapesJS loaded, but showed a blank canvas or its default "Hello world" content instead of the NovaGadget app.

**Why:** When you pass `components` and `style` to `grapesjs.init()` as strings, GrapesJS parses them. But if the HTML is complex (which ours is — 750 lines of body HTML), GrapesJS can fail silently or misparse. Also, early implementations tried to pass HTML directly via the config object, which didn't work reliably with the preset-webpage plugin.

**Fix:** Instead of passing HTML via the init config, we set the container's `innerHTML` first, then tell GrapesJS to read from the element:
```js
document.getElementById('gjs').innerHTML = rendered.bodyHTML;

gjsEditor = grapesjs.init({
    container: '#gjs',
    fromElement: true,  // ← read HTML from the #gjs div, don't use config
    // ...
});
```
This approach is more reliable because GrapesJS reads and parses the actual DOM rather than a string.

### Bug 3: Exported File Has No Working JavaScript (Broken Buttons/Navigation)

**What happened:** The user clicks Preview or Export, and the resulting page looks right but no buttons work. Clicking "Shop" in the nav does nothing. "Add to Cart" does nothing. Modals don't open.

**Why:** Two separate issues:

1. GrapesJS strips all `onclick`, `onsubmit`, `onchange`, etc. attributes when it parses HTML. These are inline event handlers that make buttons work. GrapesJS removes them because it's a visual editor — it doesn't want JavaScript executing in the editing canvas.

2. The initial `reassemble()` tried to use `gjsEditor.getHtml()` as the HTML source. But GrapesJS's HTML output has other problems too — it restructures elements, removes SVGs that were inside buttons, adds wrapper elements, and loses dynamic content that JavaScript had injected.

**Fix — Attempt 1 (data-biz-id map and restore):** Before loading into GrapesJS, we tag every element that has inline handlers with a `data-biz-id` attribute and store the handler values in a map. After export, we find elements by `data-biz-id` and re-apply their handlers. This worked for the handlers but didn't fix the other HTML mangling issues.

**Fix — Attempt 2 (current solution — use original file as output):** We stopped using `gjsEditor.getHtml()` entirely. The `reassemble()` function returns the **original source HTML** (`savedSourceHTML`) with GrapesJS CSS overrides injected as an additional `<style>` block. This means:
- All 74 inline handlers are present (they were never removed from `savedSourceHTML`)
- All SVGs are intact
- All dynamic content containers are intact
- All original CSS is intact
- The user's visual edits are layered on top as CSS overrides

### Bug 4: CDN Loading Failures

**What happened:** The editor sometimes failed to load with network errors for GrapesJS and its plugins.

**Why:** Initially, the CDN URLs pointed to unpkg.com, which had reliability issues (503 errors, slow response times, timeouts).

**Fix:** Switched all CDN URLs to jsDelivr, which is more reliable:
```html
<script src="https://cdn.jsdelivr.net/npm/grapesjs@0.21.13/dist/grapes.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/grapesjs-preset-webpage@1.0.3/dist/index.js"></script>
<script src="https://cdn.jsdelivr.net/npm/grapesjs-blocks-basic@1.0.2/dist/index.js"></script>
<script src="https://cdn.jsdelivr.net/npm/grapesjs-plugin-forms@2.0.6/dist/index.js"></script>
<script src="https://cdn.jsdelivr.net/npm/grapesjs-style-bg@2.0.2/dist/index.js"></script>
```

### Bug 5: CSS Missing After Export (Broken Layout, No Animations, No Responsive)

**What happened:** The exported file looked completely different from the original — no responsive breakpoints, no animations, no hover effects, no modal positioning.

**Why:** `gjsEditor.getCss()` only returns CSS rules that GrapesJS itself created or tracks. It does NOT return the original CSS that was loaded into the canvas. So if the original file had 390 lines of CSS and the user only changed one element's color, `getCss()` returns ~3 lines, not 390.

**Fix — evolution:**

First attempt: Concatenate `savedOriginalCSS + gjsEditor.getCss()` and put both in one `<style>` block. This worked but was complex and had selector conflict issues.

Current approach: Since `reassemble()` now returns the original file (which already has all its CSS), we just inject GrapesJS overrides as a separate `<style>` block in `<head>`. Original CSS comes first (already in the file), overrides come second (injected), so CSS specificity naturally applies the user's changes on top.

## Technical Decisions and Why

### Why an iframe for rendering instead of just DOMParser?

`DOMParser` gives you a static DOM tree — JavaScript doesn't execute. Our source file has JavaScript that populates the product grid, builds modals, and sets up dynamic content. With just DOMParser, the GrapesJS canvas would show empty containers where products should be.

The iframe approach writes the full HTML into a real browser context. The JavaScript runs, populates the DOM, and then we extract the **rendered** result. The user sees the app as it actually looks, not just its static skeleton.

The iframe is invisible (`position: fixed; left: -9999px`) and is removed after extraction.

### Why 600ms delay after iframe write?

`iDoc.write()` + `iDoc.close()` triggers HTML parsing and script execution synchronously, but rendering (layout, image loading) may need additional event loop ticks. 600ms gives the browser time to:
- Execute all inline scripts
- Fire `DOMContentLoaded`
- Process any `setTimeout(fn, 0)` calls in the source JS
- Let the DOM settle

There's also a 5-second hard fallback timeout in case something blocks.

### Why `var` instead of `let`/`const`?

The editor uses `var` throughout instead of modern `let`/`const`. This is intentional — the editor needs to work in the broadest range of browsers possible since it's a standalone tool. `var` has universal support. The generated apps (like NovaGadget) use `const`/`let`/arrow functions because they're modern apps, but the editor itself stays conservative.

### Why string concatenation instead of template literals in editor JS?

Template literals (`` ` ``) inside a `<script>` block are dangerous when the string content contains HTML tags. The browser's HTML parser runs before the JavaScript parser. If a template literal contains `</script>`, the HTML parser sees it as the end of the script block — it doesn't know it's inside a JavaScript string.

String concatenation (`'<' + '/script>'`) avoids this because the HTML parser never sees a complete `</script>` sequence.

This is specifically why lines like:
```js
var overrideStyle = '\n<' + 'style>\n/* ... */\n' + gjsCSS + '\n<' + '/style>';
```
look odd — it's not a stylistic choice, it's a necessity.

### Why `storageManager: false` in GrapesJS init?

GrapesJS has a built-in storage manager that auto-saves to localStorage. We disabled it because:
1. We handle saving ourselves (with our own localStorage keys that include the handler map)
2. The auto-save can conflict with our "original file as source of truth" approach
3. We want explicit save (Ctrl+S or button click), not auto-save

### Why are plugins registered by string name, not by reference?

```js
plugins: ['grapesjs-preset-webpage', 'gjs-blocks-basic', ...]
```

When loading GrapesJS and plugins via CDN `<script>` tags (not ES modules), each plugin registers itself globally under a string name. You reference them by that registered name. If you were using ES modules/imports, you'd pass the actual function reference.

## Known Limitations

1. **No JavaScript editing** — The editor is CSS/HTML visual only. You cannot change JavaScript logic, add event handlers to new elements, or modify the product data array.

2. **Dynamically populated containers** — If JavaScript populates a `<div>` via `innerHTML` (like `#productGrid` or `#cartItems`), deleting that div in the editor will break the app. The protection system catches elements referenced by `getElementById` but can miss containers that are only referenced indirectly.

3. **`data-biz-id` attributes in export** — The exported file contains `data-biz-id="biz-0"` etc. on elements that had inline handlers. These are invisible to users but technically present in the HTML. They are harmless.

4. **New blocks have no functionality** — Elements dragged in from the blocks panel (text, images, columns) are purely visual. They have no JavaScript behavior. To make a new button do something, you'd need to manually add JavaScript.

5. **CSS override specificity** — In rare cases, GrapesJS may generate CSS selectors that don't override the original styles due to specificity rules. The current approach (overrides in a separate `<style>` block after the original) works for most cases because later stylesheets win on equal specificity, but deeply nested or highly specific original selectors may not be overridden.

6. **Large files (5000+ lines)** — GrapesJS canvas performance degrades with very large component trees. The current 2105-line file works fine, but significantly larger generated apps may be slow.

## How to Run Locally

```bash
# Any HTTP server works. file:// won't work because fetch() requires HTTP.

# Python
cd /path/to/nova3 && python3 -m http.server 8080

# Node
cd /path/to/nova3 && npx serve .

# PHP
cd /path/to/nova3 && php -S localhost:8080

# Then open http://localhost:8080/editor.html
```

## Future: Laravel Integration

When integrating into the Laravel app:

1. **Loading:** Replace `fetch('index.html')` with `fetch('/api/projects/' + projectId + '/html')` to load HTML from the database
2. **Saving:** Replace `handleSave()` to `PUT` the reassembled HTML back to the API instead of localStorage
3. **Serving:** Either inline editor.html into a Blade template or serve it in an iframe
4. **CSRF:** Add `X-CSRF-TOKEN` header from `meta[name="csrf-token"]` to save requests

The editor is self-contained with no build step — it can be dropped into any web server as-is.
